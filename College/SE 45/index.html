Software Design Overview

Software design is the art and science of creating a blueprint for software systems that are maintainable, scalable, and efficient.

What is Design?

Design involves creating a detailed plan of how a software system will be structured and how it will function. It bridges the gap between requirements and implementation.

Definition: Software design is the process of creating an architecture and detailed design that meets the specified requirements, considering constraints such as performance, scalability, maintainability, and user experience.

Importance of Software Design

  • Visibility: Makes the invisible visible - helps teams understand complex systems
  • Quality Assurance: Ensures adherence to quality standards from the beginning
  • Cost Reduction: Catching design flaws early reduces development and maintenance costs
  • Communication: Provides a common understanding among team members and stakeholders
  • Reusability: Well-designed components can be reused across projects

Levels of Design

  • Architecture Design: High-level system structure and component organization
  • Detailed Design: Low-level design of individual components and modules

Software Design Process

The design process follows a systematic approach to create software architectures and detailed designs.

Design Process Steps

  1. Requirements Analysis: Understand and analyze functional and non-functional requirements
  2. Architectural Design: Define the overall system structure and components
  3. Detailed Design: Design individual modules and their interactions
  4. Design Documentation: Create comprehensive design documentation
  5. Design Review: Review design against requirements and standards
  6. Design Validation: Ensure design can be implemented and meets requirements

Design Principles

  • Modularity: Divide system into independent, manageable modules
  • Abstraction: Hide complexity and expose only necessary interfaces
  • Cohesion: Elements within a module should be closely related
  • Coupling: Minimize dependencies between modules
  • Separation of Concerns: Each module should have a single, well-defined responsibility
Key Principle: High cohesion and low coupling lead to maintainable and flexible designs.

Categories of Architectural Styles

Architectural styles are reusable patterns that guide high-level system structure decisions.

1. Layered (N-tier) Architecture

Description: Organizes the system into horizontal layers, each with a specific role (presentation, business logic, data access).
  • Advantages: Easy to organize teams, clear separation of concerns, simple to understand
  • Disadvantages: Can become monolithic, performance overhead from layer traversal

2. Event-Driven Architecture

Description: Components interact through events. An event source generates events that are consumed by event subscribers.
  • Advantages: Decoupled components, highly responsive, scales well
  • Disadvantages: Complex testing, difficult to trace execution flow

3. Microservices Architecture

Description: Large applications are composed of small, independent services that can be developed and deployed separately.
  • Advantages: Independent deployment, technology diversity, scalability
  • Disadvantages: Distributed system complexity, network latency, data consistency challenges

4. Model-View-Controller (MVC)

Description: Separates application into three components: Model (data), View (UI), and Controller (logic).
  • Advantages: Separation of concerns, parallel development, easier testing
  • Disadvantages: More complexity for simple applications

5. Service-Oriented Architecture (SOA)

Description: Services are modular business capabilities exposed through standardized interfaces, typically using web services.
  • Advantages: Reusability, interoperability, flexibility
  • Disadvantages: Performance overhead, complexity, requires mature processes

Principles of Designing Class-Based Components

Object-oriented design focuses on creating well-designed classes that are modular, reusable, and maintainable.

Key Design Principles

1. Single Responsibility Principle (SRP)

Definition: A class should have one, and only one, reason to change. Each class should have a single responsibility.
Example: A User class should only handle user data, not authentication logic. Authentication should be a separate class.

2. Open/Closed Principle (OCP)

Definition: Classes should be open for extension but closed for modification.
Example: Use inheritance and interfaces to extend functionality without modifying existing code.

3. Liskov Substitution Principle (LSP)

Definition: Objects of a superclass should be replaceable with objects of its subclasses without breaking the application.

4. Interface Segregation Principle (ISP)

Definition: Clients should not be forced to depend on interfaces they do not use. Create specific interfaces rather than general ones.

5. Dependency Inversion Principle (DIP)

Definition: High-level modules should not depend on low-level modules. Both should depend on abstractions.

Class Design Best Practices

  • Cohesion: Keep related functionality together in a class
  • Encapsulation: Hide internal implementation details, expose only necessary public interfaces
  • Abstraction: Define clear contracts through interfaces and abstract classes
  • Inheritance: Use inheritance for "is-a" relationships, favor composition for "has-a" relationships
  • Polymorphism: Design for polymorphism to enable flexible and extensible designs

Design Patterns

Common design patterns for class-based components:

  • Singleton: Ensure a class has only one instance
  • Factory: Create objects without specifying exact classes
  • Observer: Define one-to-many dependencies between objects
  • Strategy: Define a family of algorithms and make them interchangeable
  • Decorator: Add behavior to objects dynamically

Software Testing Overview

Testing is a critical activity that ensures software quality and meets specified requirements.

What is Software Testing?

Definition: Software testing is the process of executing a software application or system with the intent of finding and reporting defects, and validating that it performs according to specifications.

Objectives of Testing

  • Identify defects and errors in the software
  • Verify that the software meets specified requirements
  • Validate that the software is fit for use
  • Ensure quality and reliability of the system
  • Build confidence in the software system

Testing Levels

  • Unit Testing: Testing individual units or components in isolation
  • Integration Testing: Testing how different components work together
  • System Testing: Testing the complete integrated system
  • Acceptance Testing: Testing whether the system meets user requirements and acceptance criteria

Testing Types

  • Functional Testing: Verifies that functions work as intended
  • Non-functional Testing: Tests performance, security, usability, reliability, etc.
  • Regression Testing: Ensures that changes don't break existing functionality
  • Smoke Testing: Quick sanity check that critical functionality works

White Box Testing

Also known as glass box, clear box, or structural testing.

Definition

White Box Testing: A testing approach where the tester has knowledge of the internal code structure, design, and implementation of the software. Tests are derived from code structure analysis.

Key Characteristics

  • Tests are based on internal code structure and logic
  • Requires knowledge of programming languages and code implementation
  • Testers have access to source code
  • Focuses on testing code paths and branches

White Box Testing Techniques

1. Statement Coverage

Ensuring every statement in the code is executed at least once.

Example: If code has 100 lines, aim to execute all 100 lines during testing.

2. Branch Coverage

Ensuring every branch (if/else paths) is executed at least once.

Example: For an if-else statement, test both the true and false conditions.

3. Path Coverage

Ensuring every possible path through the code is executed.

4. Condition Coverage

Ensuring every condition in a decision statement evaluates to both true and false.

Advantages of White Box Testing

  • Thorough testing of code logic and structure
  • Identifies unused code and dead paths
  • Tests internal logic and algorithms
  • Can be automated easily

Disadvantages of White Box Testing

  • Requires knowledge of programming languages
  • Time-consuming to write and maintain tests
  • May miss user-centric issues
  • Cannot test non-functional requirements easily

Black Box Testing

Also known as behavioral testing or functional testing.

Definition

Black Box Testing: A testing approach where the tester does not have knowledge of the internal code structure or implementation. Tests are based on specifications and requirements, treating the system as a "black box."

Key Characteristics

  • Tests are based on specifications and requirements
  • No knowledge of internal code structure required
  • Tester focuses on inputs and outputs
  • Tests user-visible functionality

Black Box Testing Techniques

1. Equivalence Partitioning

Dividing input data into groups that are expected to behave similarly.

Example: For age input (1-100 valid, <1 or >100 invalid), test valid range (e.g., 50) and invalid ranges (e.g., -1, 150).

2. Boundary Value Analysis

Testing values at the boundaries of input domains.

Example: For age input 1-100, test 0, 1, 100, and 101.

3. Decision Table Testing

Creating test cases for all combinations of conditions.

4. State Transition Testing

Testing the transitions between different states of the system.

Advantages of Black Box Testing

  • Tests from user perspective
  • No knowledge of programming language required
  • Identifies missing functionality
  • Tests complete system integration
  • Can be performed by non-technical testers

Disadvantages of Black Box Testing

  • May not cover all code paths
  • Tests are often redundant
  • Cannot identify specific internal bugs
  • May not test all boundary conditions

Verification and Validation (V&V)

Two critical concepts in software quality assurance that are often confused but have different meanings.

Verification

Definition: Verification is the process of evaluating whether the software product meets the specifications that were used to design it. It answers: "Are we building the product right?"
  • Focus: Internal correctness
  • Question: Does the code meet design requirements?
  • Activities: Reviews, inspections, unit testing, integration testing
  • Timing: During development

Validation

Definition: Validation is the process of establishing whether the completed software product satisfies user requirements. It answers: "Are we building the right product?"
  • Focus: Meeting user needs
  • Question: Does the system meet user requirements?
  • Activities: User acceptance testing, system testing, beta testing
  • Timing: After development, before release

V&V Process

  1. Requirements Review: Verify requirements are clear and complete
  2. Design Review: Verify design satisfies requirements
  3. Code Review: Verify code implements design correctly
  4. Unit Testing: Verify individual units work correctly
  5. Integration Testing: Verify components work together
  6. System Testing: Validate system meets requirements
  7. User Acceptance Testing: Validate system meets user needs
Key Point: Verification ensures the development process is correct; Validation ensures the final product is correct for the intended use.

Reliability Metrics and Quality Characteristics

Quantifiable measures to assess and monitor software quality and reliability.

What is Reliability?

Definition: Reliability is the probability that a system will perform its intended function without failure for a specified period of time under specified conditions.

Key Reliability Metrics

1. Mean Time Between Failures (MTBF)

Definition: Average time elapsed between system failures. Calculated as: Total service time / Number of failures.
Example: If a system runs for 10,000 hours with 10 failures, MTBF = 1,000 hours.

2. Mean Time To Repair (MTTR)

Definition: Average time taken to fix a failure and restore the system to operation.

3. Availability

Definition: Percentage of time a system is available and usable. Calculated as: MTBF / (MTBF + MTTR)
Example: MTBF = 100 hours, MTTR = 2 hours, Availability = 100/(100+2) = 98.04%

4. Failure Rate

Definition: Number of failures per unit time. Used to track system stability.

5. Defect Density

Definition: Number of defects per unit size (e.g., defects per 1000 lines of code - KLOC).

Software Quality Characteristics

Product Operation Qualities

  • Correctness: Extent to which program satisfies specifications and fulfills user objectives
  • Reliability: Ability to perform intended function with required precision
  • Efficiency: Amount of computing resources required by software
  • Integrity: Extent to which access to software/data by unauthorized persons is controlled
  • Usability: Effort required to learn, operate, and interpret output

Product Revision Qualities

  • Maintainability: Effort required to locate and fix errors
  • Testability: Effort required to test that system performs intended function
  • Flexibility: Effort required to modify operational program

Product Transition Qualities

  • Portability: Effort required to transfer program to different hardware/software
  • Reusability: Extent to which program can be used in other applications
  • Interoperability: Effort required to couple one system to another

Quality Models

McCall's Model: Quality defined in terms of operational characteristics, maintainability, and portability.

Boehm's Model: Quality based on as-is utility, maintainability, and portability.

ISO 9126 Model: Defines quality in terms of functionality, reliability, usability, efficiency, maintainability, and portability.

Reviews and Inspections

Systematic processes for evaluating software artifacts to ensure quality and identify defects early.

What are Reviews and Inspections?

Definition: Reviews and inspections are static testing techniques where software artifacts (documents, code, design) are examined without execution to identify defects and ensure compliance with standards.

Types of Reviews

1. Formal Inspection

A rigorous, structured review process with defined roles and procedures.

  • Moderator: Leads the inspection process
  • Author: Creator of the artifact being reviewed
  • Reviewers: Examine the artifact for defects
  • Recorder: Documents defects and issues found

2. Walk-through

An informal review where the author explains the artifact to reviewers.

  • Less structured than formal inspection
  • Author leads the review
  • Good for knowledge transfer and team understanding

3. Code Review

Systematic examination of source code by peers.

  • Identifies coding errors and violations of standards
  • Improves code quality and maintainability
  • Can be formal or informal peer reviews

Inspection Process Steps

  1. Planning: Prepare for inspection, identify participants
  2. Overview: Introduce the artifact and inspection procedures
  3. Preparation: Participants review artifact individually
  4. Meeting: Group discussion to identify and record defects
  5. Rework: Author fixes defects found
  6. Follow-up: Verify rework and close inspection

Benefits of Reviews and Inspections

  • Find defects early in development (before testing)
  • Cost-effective: cheaper to fix defects earlier
  • Improve code quality and maintainability
  • Promote knowledge sharing among team members
  • Establish coding standards and best practices
  • Build team morale and collaboration

Best Practices

  • Define clear inspection criteria and checklists
  • Keep inspection meetings focused and time-bound
  • Separate defect finding from defect fixing
  • Document all defects and their resolutions
  • Use metrics to track inspection effectiveness
  • Make inspections a regular part of development process

Quality Models

Quality models provide frameworks for understanding and measuring software quality.

McCall's Quality Model

Overview: McCall defined quality in terms of three high-level parameters and eleven quality factors.

Three High-Level Parameters:

  • Operational Characteristics: How well the software operates
  • Maintenance Characteristics: How easy it is to fix defects
  • Transition Characteristics: How easy it is to port to different platforms

Eleven Quality Factors: Correctness, Reliability, Efficiency, Integrity, Usability, Maintainability, Flexibility, Testability, Portability, Reusability, Interoperability

Boehm's Quality Model

Overview: Boehm suggested quality based on three high-level characteristics important to users.
  • As-is Utility: How well (easily, reliably, efficiently) it can be used
  • Maintainability: How easy to understand, modify, and retest
  • Portability: Difficulty in making software work in changed environment

ISO 9126 Quality Model

Overview: International standard for software quality characteristics.

Six Quality Characteristics:

  • Functionality: Ability to provide functions that meet stated needs
  • Reliability: Ability to maintain level of performance
  • Usability: Effort needed for use and individual assessment
  • Efficiency: Performance in relation to resources
  • Maintainability: Effort needed to make modifications
  • Portability: Ability to be transferred to different environments

Garvin's Eight Quality Dimensions

  • Performance: How well it performs the jobs
  • Features: How well it supports required features
  • Reliability: Probability of working satisfactorily within specified time
  • Conformance: Degree to which product meets requirements
  • Durability: Measure of product life
  • Serviceability: Speed and effectiveness of maintenance
  • Aesthetics: The look and feel of the product
  • Perceived Quality: User's opinion about product quality
